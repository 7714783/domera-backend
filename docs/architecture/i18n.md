# i18n — interface language, not content language

**Status:** Foundation + Phase 1/2 complete. Phase 3/4 string-migration is incremental per module, tracked in the developer dashboard.

## Non-negotiables

- Supported UI locales: **en**, **ru**, **he**.
- Default + fallback: **en**. When a key is missing in `ru` / `he`, the UI still renders (backstop = empty string or English source, depending on loader).
- **UI language ≠ business content language.** Switching the UI from RU to HE must not translate a building's name, a unit label, or a user-authored note. Those are data.
- **No hardcoded user-facing strings** in screens, forms, or shared components. Everything user-visible routes through the translation tree.
- `he` triggers RTL layout (web `<html dir>`, mobile `I18nManager.forceRTL`).

## Where translations live

### Web (`apps/frontend/src/i18n/`)

```
i18n/
├── config.ts              # LOCALES, NAMESPACES, dirFor(), LOCALE_STORAGE_KEY
├── load.ts                # server-side dictionary loader
├── LocaleProvider.tsx     # client-side t(ns.key) provider
├── LocaleDirApplier.tsx   # applies dir='rtl' | 'ltr' on <html>
├── formatters.ts          # formatDateByLocale / formatCurrencyByLocale / …
├── persistence.ts         # readPersistedLocale / writePersistedLocale / resolveInitialLocale
└── locales/
    ├── en/{common,nav,shell,auth,onboarding,dashboard,buildings,ppm,assets,
    │        cleaning,documents,team,qr,forms,validation,errors,statuses,settings,mobile}.json
    ├── ru/…               # same 19 files — structure enforced by i18n:check
    └── he/…
```

### Mobile (`apps/mobile/src/i18n/`)

Single TS file per locale with `typeof en`-typed `ru` and `he` — the TS compiler refuses to merge if key trees drift. Persistence uses `expo-secure-store`. RTL toggled via `I18nManager` on `he`.

## The rule for components

```tsx
// ❌  hardcoded
<Button>Save</Button>

// ✅  translated
const t = useTranslations('forms');
<Button>{t('actions.save')}</Button>
```

For the `shared/ui/*` primitives:

- Accept a `title`, `label`, `placeholder` as props. Don't default them to English — let the caller localize.
- If a component truly has a fixed domain label (e.g. "loading…"), import it from `common` or `forms`, not hardcoded.

## Priority chain for initial locale

Evaluated in this order:

1. **URL segment** `/{locale}/...` — always wins. Deep links share a specific locale.
2. **Persisted preference** (`localStorage.getItem(LOCALE_STORAGE_KEY)` on web, `secure-store` on mobile).
3. **Browser / device language**.
4. **Default `en`**.

Helper: `resolveInitialLocale(input)` in `i18n/persistence.ts`.

## RTL

### Web
- `<html dir="rtl" | "ltr">` set by `LocaleDirApplier`. App already uses Tailwind logical properties where possible; legacy absolute `left`/`right` styling is flagged by code review.

### Mobile
- `I18nManager.allowRTL()` + `I18nManager.forceRTL()` triggered when locale changes.
- RN requires a **full app restart** to apply `forceRTL`. The caller of `setLocale('he')` should surface a "Restart app" banner. This is a React Native platform constraint, not our choice.

## Formatters

Never `.toLocaleDateString()` ad hoc. Use:

```ts
import { formatDateByLocale, formatCurrencyByLocale } from '@/i18n/formatters';

formatDateByLocale(task.dueAt, locale);       // 'Apr 30, 2026'   /   '30 апр. 2026 г.'   /   '30 באפר׳ 2026'
formatCurrencyByLocale(5000, locale, 'ILS');  // '₪5,000.00'      /   '5 000,00 ₪'        /   '‏5,000.00 ₪‎'
```

## Status + enum translation

Enum values live in the DB in English (`pending`, `in_progress`, `completed`…). The UI translates via `statuses.ppmLifecycle.in_progress` etc.

Rule: never branch UI on a translated string. Branch on the enum, translate at render.

## Tooling

### `i18n:check` (hard gate in CI)

```
pnpm run i18n:check
```

Flattens every namespace in every locale and asserts the **same leaf-key set**. Fails if a key exists in `en` but not in `ru` — or vice versa. Runs on every PR; structural drift can't land on `main`.

### `i18n:audit` (warn-only)

```
pnpm run i18n:audit
```

Greps JSX/TSX for suspicious hardcoded UI strings (JSX text between tags starting with an uppercase letter, `placeholder="..."`, `title="..."`). False positives happen — the exit is always 0. Use it as a hint list during code review.

Opt-out per file with a top-of-file `// i18n:ignore` comment.

## Adding a new translation key

1. Decide which namespace fits. If none fits, add one to `i18n/config.ts::NAMESPACES` and create the three JSON files (en/ru/he).
2. Add the key in `en/<ns>.json`.
3. Copy the same key to `ru/<ns>.json` and `he/<ns>.json`. Use the English string as a placeholder if a translation isn't ready — `i18n:check` cares about **keys**, not values.
4. Reference with `useTranslations('ns').('path.to.key')`.
5. `pnpm run i18n:check` locally before pushing.

## Adding a new language

1. Add the ISO code to `LOCALES` in `i18n/config.ts`.
2. Create the `locales/<code>/` folder + mirror every namespace file from `en/`.
3. Add `<code>` to `RTL_LOCALES` if it's an RTL language.
4. Add `LOCALE_LABEL[<code>]`.
5. Mobile: add a `<code>.ts` file in `apps/mobile/src/i18n/locales/` typed as `typeof en`.
6. `pnpm run i18n:check` to confirm.

## Anti-patterns (caught in review)

- `<p>{`Welcome, ${name}`}</p>` — use an interpolation key.
- `if (status === 'In progress')` — branch on enum, translate at render.
- `getStatusLabel(status: string): string` that hardcodes switch-case English strings.
- Keeping Russian strings inside the `ru` profile page while the rest of the page pulls from `en`. Everything under one locale.
- Storing translated strings in the database as data (breaks "UI lang ≠ content lang").
